## 1. Суть проекту

Каса (POS) для магазину спортивного харчування **«Геркулес Шоп»**
у вигляді PWA-додатку: працює **офлайн**, дані зберігає **локально (IndexedDB)**,
товари **читає з CSV-файлу за постійним шляхом** (`public/catalog.csv`, редагується в Excel).

Принцип: **локальна база — єдине джерело правди. Ніякої хмари, ніякого OAuth.**
`catalog.csv` — основний і незмінний каталог: жодної кнопки імпорту, каса читає
його автоматично при кожному запуску.

```
public/catalog.csv (Excel) ──автозчитування при запуску──▶ IndexedDB ◀──▶ Інтерфейс каси
```

> Було: Google Sheets API v4 (імпорт) + Google Drive API (бекап чеків). Обидва
> прибрані — код був не підключений в UI (нема OAuth), фактично не працював.
> З 2026-07-05: бекап є, але локальний і ручний — кнопки «Завантажити бекап» /
> «Відновити з файлу» на екрані «Товари» вивантажують/відновлюють JSON-файл
> з усією базою (категорії + товари + чеки). Використовується для перенесення
> каси на інший ПК. Без цього файлу — дані живуть лише в IndexedDB браузера.

## 3. КОНТРАКТ ДАНИХ (головне! всі модулі використовують ТІЛЬКИ ці формати)

### 3.1 Товар (product)
```js
{
  id: 1,                    // number, унікальний, з catalog.csv
  cat: "protein",           // string, id категорії
  name: "Whey Protein 900г ваніль",  // string, назва + фасування + смак
  price: 1250,              // number, грн, ціна продажу
  cost: 850,                // number, грн, собівартість (з колонки E catalog.csv; 0, якщо не задано — товари, додані вручну через «Товари», собівартості не мають)
  stock: 12,                // number, поточний залишок (шт/упаковок)
  updatedAt: 1720000000000  // number, timestamp останньої зміни
}
```

### 3.2 Категорія (category)
```js
{ id: "protein", name: "Протеїни", emoji: "🥛", order: 1 }
```

Орієнтовні категорії магазину (фінальний список — у `catalog.csv`):

| id        | Назва                  | emoji |
|-----------|------------------------|-------|
| protein   | Протеїни               | 🥛    |
| gainer    | Гейнери                | 🏋️    |
| creatine  | Креатин                | ⚡    |
| amino     | Амінокислоти / BCAA    | 🧬    |
| preworkout| Передтренувальні       | 🔥    |
| vitamins  | Вітаміни / мінерали    | 💊    |
| bars      | Батончики / снеки      | 🍫    |
| accessories| Аксесуари (шейкери…)  | 🥤    |

### 3.3 Чек (receipt)
```js
{
  no: 1,                    // number, наскрізний номер за зміну
  time: 1720000000000,      // timestamp створення
  items: [                  // позиції на момент продажу (ціни фіксуються!)
    { id: 1, name: "Whey Protein 900г ваніль", price: 1250, qty: 1 }
  ],
  subtotal: 1250,           // сума без знижки
  discount: 10,             // знижка у %
  total: 1125,              // до сплати
  cancelled: false          // true = сторно (чек НЕ видаляється!)
}
```

## 4. API між модулями (сигнатури функцій — не змінювати)

### Модуль бази (db.js) — Чат 1
```js
initDB()                        // → Promise<void>
getCategories()                 // → Promise<category[]>
getProducts(catId?)             // → Promise<product[]>
searchProducts(query)           // → Promise<product[]>  // пошук за назвою
createProduct({cat,name,price,stock}) // → Promise<product>  // id автоінкремент з 900000
deleteProduct(id)                // → Promise<void>  // реальне видалення, без сторно
createReceipt(items, discount)  // → Promise<receipt>  // списує stock
cancelReceipt(no)                // → Promise<receipt>  // повертає stock, ставить cancelled
getReceipts()                    // → Promise<receipt[]>
getDayTotal()                    // → Promise<{sum, count}>
getTodayStats()                  // → Promise<{sum,count,avgCheck,discountTotal,cancelledCount}>  // дешево, індексований запит за сьогодні
getFullStats()                   // → Promise<{allTime, stock, topProducts}>  // повний прохід, оновлювати рідше
getStats()                       // → Promise<{today, allTime, stock, topProducts}>  // getTodayStats+getFullStats разом, для одноразового виклику
exportBackup()                   // → Promise<{version, exportedAt, categories, products, receipts}>
importBackup(backup)             // → Promise<void>  // повністю замінює базу вмістом бекапу
```

### Модуль імпорту каталогу (sync.js) — Чат 2
```js
parseCatalogCSV(csvText)  // → {categories, products} — розбір catalog.csv
importFromFile(csvText)   // CSV → IndexedDB, конфлікти вирішує updatedAt
```

### Інтерфейс (Чат 3) викликає ТІЛЬКИ ці функції, напряму в БД не лізе.

## 5. Структура public/catalog.csv (генерується `scripts/migrate-woo.mjs`, редагується в Excel)

Файл лежить за постійним шляхом `public/catalog.csv` → на запущеній касі доступний
за URL `/catalog.csv`. Каса (`App.jsx`) при кожному старті сама робить `fetch('/catalog.csv')`
і імпортує його в IndexedDB (`importFromFile`) — кнопки ручного імпорту немає.

Один файл, дві секції через порожній рядок:

```
id,назва,emoji,порядок
protein,"Протеїни",🥛,1
...

id,категорія,назва,кількість,собівартість,ціна продажу
1,protein,"Whey Protein 900г ваніль",12,,1250
...
```

**Колонка E (собівартість)** — читається в поле `product.cost` і використовується
в розділі «Статистика» (вартість залишку за собівартістю, потенційний прибуток).
Файл має бути в **UTF-8 з BOM** (інакше Excel на Windows зіпсує кирилицю) —
`migrate-woo.mjs` вже пише BOM автоматично.

**Правило назв:** назва містить фасування і смак («900г ваніль»), бо один
продукт у різних вагах/смаках — це РІЗНІ товари з різними id і залишками.

## 7. Точки тестування

- **🧪 Т1 (після Чату 1):** чек через консоль → stock списався; сторно → повернувся; перезавантаження → дані живі.
- **🧪 Т2 (після Чату 2):** імпорт реального `catalog.csv`; зміна ціни/залишку у файлі підтягується при повторному імпорті, локальний stock не втрачається.
- **🧪 Т3 (після Чату 3):** повний цикл на екрані (продаж → знижка → оплата → журнал → сторно) + пошук товару + перевірка на планшеті.
- **🧪 Т4 (Чат 4, приймальний):** робочий день з реальними товарами, навмисний обрив Wi-Fi посеред зміни (додаток офлайн-first, мережа й так не потрібна для роботи каси), звірка залишків з `catalog.csv` ввечері.

**Правило збірки:** Чат 4 стартує лише після зелених Т1–Т3.

## 8. Правила для всіх чатів

1. Формати даних і сигнатури з розділів 3–4 — **закон**. Зміна = оновити цей паспорт і повідомити в інші чати.
2. Скасований чек — це сторно (`cancelled: true`), ніколи не видалення.
3. Ціна в чеку фіксується на момент продажу (зміна ціни товару не змінює старі чеки).
4. Кожна зміна даних оновлює `updatedAt` — це основа вирішення конфліктів синхронізації.
5. Інтерфейс: великі кнопки під палець, **пошук за назвою** (асортимент більший, ніж у кафе), мінімум анімацій, швидке завантаження.
6. Один продукт у різних фасуваннях/смаках = різні товари з різними id.
7. Брендинг з розділу 0 використовується всюди: шапка каси, чек, manifest PWA.
